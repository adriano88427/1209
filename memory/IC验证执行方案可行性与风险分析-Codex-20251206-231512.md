# IC验证执行方案可行性与风险分析

## 可行性
- 技术可行：现有代码已支持差异表/审计摘要生成，新增开关默认关闭，不影响主路径；开启后仅生成附加输出。
- 数据量可控：当前约 2 万行、1148 交易日，差异/审计计算为 O(N)，性能风险低；基准/CI 计划只对核心因子，成本可控。
- 配置简洁：关键开关集中在 `fa_config`，默认关闭，CLI 仅需 `--run-ic-validation` 即可触发验证。
- 基线验证：已跑通默认配置，IC 差异 0，证明基线保持不变。

## 风险点与缓解
1) 零/负价列匹配
   - 风险：配置列名与真实数据不符，计数失效或误判。
   - 缓解：确认/调整 `ZERO_PRICE_COLUMNS`；默认 `ZERO_PRICE_ACTION=off` 不改数据。
2) 审计输出/路径
   - 风险：latest_summary_path 为空时跳过验证；输出文件命名不一致。
   - 缓解：已改为 `audit_summary_latest.csv`；验证前确认汇总路径可用，否则提示跳过。
3) 覆盖提示
   - 风险：开启时新增列影响下游解析。
   - 缓解：`ENABLE_COVERAGE_HINT` 默认 False，开启后需手动验证下游。
4) 基准/CI 占位
   - 风险：误开启占位输出无意义数据。
   - 缓解：`ENABLE_IC_BENCHMARK/ENABLE_IC_CI` 默认 False，说明文档明确“核心因子 + TODO”占位，实施时再落地。
5) 性能
   - 风险：开启基准/CI 时耗时上升。
   - 缓解：仅核心因子执行，必要时采样或关闭。

## 结论
- 在默认关闭开关、按需逐项开启的前提下，方案可行且对主路径影响可控。
- 优先执行零价列确认和 `--run-ic-validation` 基础验证；基准/CI 在核心因子小范围试点后再推广。***
